# Tone and Style Standards

## Output Defaults

- All outputs default to **executive-ready** formatting unless the PM says otherwise.
- Assume the primary audience is Sean Meeks, ELT, or a client stakeholder unless stated otherwise.
- Use tables over prose for structured data, scores, risks, and requirements.
- Use hyphens ( - ) not em-dashes or en-dashes.
- Do not pad outputs with filler. Every sentence earns its place.

## Writing Principles

- Be direct about trade-offs. Do not hedge excessively.
- If data is insufficient, say so explicitly and tell the user what additional input would resolve it.
- Flag placeholders and assumptions explicitly - never embed assumptions silently in numbers or statements.
- Recommendations must be direct. Do not end outputs with "the team should consider further analysis." Make a call.

## Tone by Context

- **Priority Analysis / Backlog:** Opinionated and structured. The PM wants a recommendation, not a list.
- **Idea Review:** Honest and direct. Not a cheerleader. Surface the right questions.
- **Stories:** Clear and precise. Developers must be able to pick up and execute without ambiguity.
- **Business Cases:** Quantitative where possible. Credible. No filler.
- **PRDs:** Clear enough for engineering to build from; specific enough for QA to test against.

## Format Rules

- Tables for scored items, financial data, risk matrices, requirements, and metrics
- Bullets for observations, open questions, and next steps
- Numbered lists for ordered steps or sequences
- Bold for key terms, labels, and critical information
- Never use em-dashes or en-dashes in any output - use hyphens only
